# 💬 Блок 2: Conversation Memory

_4 стратегії керування контекстом чату_

<!--

# Стратегія 1: Buffer Memory (все підряд)

**Ідея:** зберігати ВСЮ розмову, відправляти ВСЮ в LLM.

```
Розмова: [msg1][msg2][msg3]...[msg100]
                    ↓ відправляємо в LLM
LLM отримує: [msg1][msg2]...[msg100][новий_запит]
```

```js
import { BufferMemory } from "langchain/memory";
import { ConversationChain } from "langchain/chains";
import { ChatOpenAI } from "@langchain/openai";

const llm = new ChatOpenAI({ configuration: { baseURL: process.env.LITELLM_BASE_URL } });
const memory = new BufferMemory();  // зберігаємо все

const chain = new ConversationChain({ llm, memory });

await chain.invoke({ input: "Мене звуть Іван" });
await chain.invoke({ input: "Як мене звуть?" });
// → "Вас звуть Іван ✅"
```

<!--
НОТАТКИ ДОПОВІДАЧА:
Buffer Memory — це те що робить ChatGPT за замовчуванням у межах сесії.
Зберігати все = простіше за все, але не масштабується.

Коли прийнятно: PoC, демонстрація, короткі сесії (≤20 повідомлень).
Коли небезпечно: long-running агент, задачі що тривають годинами.

При 128K контексті (GPT-4o) теоретично можна мати ~500 повідомлень по 250 токенів.
Але якість відповідей починає падати вже після 50-100 повідомлень через "lost in middle".

LiteLLM baseURL: наш proxy на localhost:4000 або за конфігурацією.
Ми підключаємо наші власні моделі (MiniMax M3 та ін.) через нього.

# Стратегія 2: Window Memory (останні N)

**Ідея:** зберігати тільки **останні N повідомлень**. Старі — викидати.

```
Розмова: [msg1][msg2][msg3][msg4][msg5][msg6]...[msg100]
                                 ↓ після 6+ повідомлень:
Window(k=4):                 [msg97][msg98][msg99][msg100][новий]
                              ↑ тільки останні 4 пари
```

```js
import { BufferWindowMemory } from "langchain/memory";

const memory = new BufferWindowMemory({ k: 4 });  // зберігаємо 4 останніх обміни
// k=4 означає 4 пари "запит + відповідь" = 8 повідомлень

const chain = new ConversationChain({ llm, memory });
```

<div>

❌ Забуває за межею вікна | "Амнезія через N повідомлень"

<!--
НОТАТКИ ДОПОВІДАЧА:
Window Memory — хороший баланс простоти і контролю для продакшн чат-ботів.
k=10-20 зазвичай достатньо для "розумної" розмови.

Реальна проблема: "Але я ж казав це 30 повідомлень тому!"
Якщо юзер посилається на давнє повідомлення — агент не пам'ятає.

Аналогія: людина з короткою пам'яттю (7 хвилин). Остання розмова пам'ятає, все до → ні.

k=4 → це 4 "туди-сюди" (user + assistant = 1 exchange).
Якщо середнє повідомлення 100 токенів → 4×2×100 = 800 токенів на history. Дешево.
-->

# Стратегія 3: Summary Memory (стискаємо через LLM)

**Ідея:** регулярно **стискати** стару розмову в короткий підсумок через LLM.

```
Розмова:    [msg1][msg2]...[msg50] | [msg51][msg52]...[msg100]
                  ↓ LLM стискає            ↓ нові (не стиснуті)
Summary:    "Іван питав про SQL.           [msg51][msg52]...[msg100]
             Знайшли помилку в JOIN-і.
             Вирішили переписати запит."
                  ↓ що відправляємо в LLM при новому запиті:
До LLM:     [Summary] + [msg51]...[msg100] + [новий_запит]
```

```js
import { ConversationSummaryMemory } from "langchain/memory";

const memory = new ConversationSummaryMemory({ llm });
// LLM сам стискає коли history стає довгою
```

<div>

✅ Не втрачає суть | Контекст не росте | Для тривалих сесій

</div>
<div>

❌ Кожне стиснення = доп. виклик LLM | Деталі можуть загубитись

</div>
</div>

<!--
НОТАТКИ ДОПОВІДАЧА:
Summary Memory — як "конспект лекції". Зберігаємо суть, конкретні деталі — ні.

ВАЖЛИВО: стиснення само по собі коштує токени!
Кожне summary = виклик LLM на 1000-2000 вхідних токенів.
Для частих розмов це може бути значна вартість.

Якість summary залежить від якості LLM і промпту для стиснення.
Гірший LLM → гірший summary → гірша пам'ять агента.

Практично: ConversationSummaryMemory в LangChain v0.3+ вважається legacy.
LangGraph + StateGraph з power summarization node — сучасна альтернатива.
Але для навчання — ConversationSummaryMemory ідеальний для розуміння концепції.
-->

# Стратегія 4: Summary Buffer (гібрид)

**Ідея:** зберігаємо **підсумок старого** + **повні останні N** повідомлень.

```
До LLM відправляємо:

┌─────────────────────────────────────────────────┐
│  SUMMARY (стиснуте резюме всього що було):      │
│  "Обговорювали SQL оптимізацію, знайшли        │
│  проблему з індексами, вирішили додати індекс"  │
├─────────────────────────────────────────────────┤
│  RECENT (останні 6 повідомлень — повністю):     │
│  [msg95][msg96][msg97][msg98][msg99][msg100]    │
└─────────────────────────────────────────────────┘
              ↓ + новий запит
```

```js
import { ConversationSummaryBufferMemory } from "langchain/memory";

const memory = new ConversationSummaryBufferMemory({
  llm,
  maxTokenLimit: 1000  // якщо перевищено → стискає старе в summary
});
```

<!--
НОТАТКИ ДОПОВІДАЧА:
SummaryBuffer — найпоширеніший вибір для продакшн асистентів.

maxTokenLimit=1000: коли history перевищує 1000 токенів → LLM стискає найстарішу частину в summary.
Нові повідомлення залишаються повністю.

Переваги:
- Пам'ятає "суть" всього що було (через summary)
- Точно пам'ятає останнє (через буфер)
- Контрольований розмір

Зазвичай для продакшн: maxTokenLimit = 2000-4000.
При MiniMax M3 256K контексті можна збільшити значно.
-->

# Порівняння 4 стратегій

| Стратегія | Як працює | Розмір контексту | Пам'ятає далеке? | Коли обирати |
|-----------|-----------|-----------------|------------------|--------------|
| **Buffer** | Все підряд | Росте ∞ | ✅ Так | PoC / тести |
| **Window** | Останні N | Фіксований | ❌ Ні | Простий продакшн |
| **Summary** | Стискає все | Малий | ✅ Суть | Тривалі сесії |
| **Summary+Buffer** | Summary + останні N | Контрольований | ✅ Суть + деталі | **Продакшн** |

```
Тривалість сесії ≤ 20 повідомлень?   → Window (k=10)
Тривала сесія (30+)?                  → SummaryBuffer (maxTokenLimit=2000)
Критична точність деталей?            → Buffer (з обмеженням через ліміт моделі)
```

<!--
НОТАТКИ ДОПОВІДАЧА:
Таблиця — швидка шпаргалка для вибору стратегії.

Найчастіший помилковий вибір: Buffer для тривалих агентів.
Правильний для більшості: SummaryBuffer.

Питання до аудиторії: "Яку стратегію ви б обрали для..."
а) Чат-підтримки клієнтів (короткі сесії, нові юзери) → Window
б) Особистого асистента (тривала сесія, знайомий юзер) → SummaryBuffer
в) Code агента (довга задача, важливі технічні деталі) → SummaryBuffer або LangGraph
-->

# ⚠️ Важливо: це пам'ять **однієї сесії**

Між сесіями — все зникає якщо не зберегти в БД.

```
Сесія 1: [розмова про SQL] → закрили браузер → зникло
Сесія 2: LLM нічого не знає про Сесію 1
```

**Для пам'яті між сесіями** використовуй LangGraph + SqliteSaver:

```js
import { SqliteSaver } from "@langchain/langgraph/checkpoint/sqlite";
import { createReactAgent } from "@langchain/langgraph/prebuilt";

// Зберігає весь стан агента в SQLite між запитами:
const checkpointer = SqliteSaver.fromConnString("./memory.db");
const agent = createReactAgent({ llm, tools, checkpointer });

const config = { configurable: { thread_id: "user_ivan_001" } };
// thread_id = унікальний ID розмови (зберігається на диску!)

await agent.invoke({ messages: [["user", "Привіт, я Іван"]] }, config);
// [перезапускаємо процес]
await agent.invoke({ messages: [["user", "Як мене звуть?"]] }, config);
// → "Вас звуть Іван ✅" — пам'ятає після перезапуску!
```

<!--
НОТАТКИ ДОПОВІДАЧА:
Дуже часта помилка початківців: думають що Summary Memory = довгострокова пам'ять.
НІ! Все що в RAM — зникає при перезапуску.

SqliteSaver: зберігає стан на диску.
PostgresSaver: для PostgreSQL (краще для масштабу).
MemorySaver: тільки в RAM (для тестів, не для проду).

thread_id — ключовий параметр:
{ thread_id: "global" }              → одна пам'ять для всіх (небезпечно!)
{ thread_id: `user_${userId}` }      → окрема пам'ять per user ← правильно
{ thread_id: `sess_${sessionId}` }   → окрема per session

Детальніше про LangGraph — в Блоці 8 (Фреймворки).
-->
