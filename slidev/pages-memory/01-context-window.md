# LLM — це функція, не база даних

**Порівняння з Google:**

```
╔══════════════════════════════════════════════════════════╗
║  Ти: "Як приготувати борщ?"                             ║
║  Google: [показує результати]                            ║
╠══════════════════════════════════════════════════════════╣
║  Завтра: "Що я питав вчора?"                            ║
║  Google: ??? — він не пам'ятає ваш конкретний запит    ║
╚══════════════════════════════════════════════════════════╝

LLM — те саме. Кожен виклик: вхідний текст → вихідний текст.
```

<v-click>

**Технічно кажучи:**
```
f(текст_на_вході) → текст_на_виході
```

Немає "пам'яті між викликами". Немає "стану". Тільки вхід і вихід.

**Але тоді як ChatGPT "пам'ятає" розмову?**
→ ChatGPT передає ВСЮ попередню розмову разом з кожним новим запитом.
→ LLM не пам'ятає — вона читає все заново кожного разу.

</v-click>

<!--
НОТАТКИ ДОПОВІДАЧА:
Аналогія з Google дуже зрозуміла для нетехнічної аудиторії.

Важливо: ChatGPT "пам'ятає" завдяки тому що браузерний код зберігає
всі повідомлення і відправляє їх ALL разом з кожним новим запитом.
Само LLM (GPT-4) не має стану між викликами.

Ще одна аналогія для програмістів:
LLM = stateless функція. Як pure function в functional programming.
Той самий вхід → завжди той самий вихід (з поправкою на temperature).

Власні моделі в нас теж працюють так само — MiniMax M3, LiteLLM proxy.
Не має значення яку модель використовуєш: всі вони stateless.

# Контекстне вікно — "аркуш паперу" LLM

**Термін: Context Window** — максимальна кількість тексту, яку LLM може прочитати за один виклик.
Вимірюється в **токенах**.

**Термін: Токен** — шматочок тексту (приблизно 0.75 слова або 4 символи).

<div grid="~ cols-2 gap-6">
<div>

```
"кіт"          → 1 токен
"неухильно"    → 2 токени
"Hello world"  → 2 токени
"привіт"       → 2-3 токени (українська = більше)
```

**Перевір свій текст:**
- `tiktoken` для OpenAI моделей
- `platform.openai.com/tokenizer` — онлайн
- Власні моделі: перевіряй через `tokenize` endpoint

</div>

# KV Cache: як моделі прискорюють повторні виклики

**Термін: KV Cache** — кешування обчислень attention для статичних частин промпту.

```
БЕЗ КЕШУ:
Запит 1: [sys 5K][history][новий]
         → рахуємо ВСЕ
Запит 2: [sys 5K][history][новий]
         → рахуємо ВСЕ знову
         ↑ sys 5K — кожен раз!

З KV CACHE:
Запит 1: [sys 5K][history][новий]
         → рахуємо все
         └─ кешуємо sys ─┘
Запит 2: [CACHED✓][history][новий]
         → рахуємо тільки нове
         ↑ sys НЕ рахується знову
         → швидше + до 90% дешевше
```

::right::

**Claude API** (`cache_control`):

```js
messages = [{
  role: "user",
  content: [{
    type: "text",
    text: великий_системний_промпт,
    cache_control: { type: "ephemeral" }
  }]
}]
// до 90% економії · ~3x швидше
```

**Що кешувати:**
промпт (правила, SOP) · схема БД · few-shot приклади

**Що НЕ кешувати:**
поточне повідомлення юзера · conversation history

<!--
НОТАТКИ ДОПОВІДАЧА:
KV Cache = Key-Value Cache від механізму self-attention в трансформерах.
При обробці тексту модель рахує "ключі" (keys) і "значення" (values) для кожного токену.
Якщо частина тексту не змінюється → можна зберегти ці обчислення.

Практично важливо коли:
1. Великий системний промпт (правила, SOP, схема БД) — 3-5K токенів
2. Документи які не змінюються (wiki, довідники)
3. Тривалі сесії з багатьма запитами

Anthropic Claude: cache_control: { type: "ephemeral" }
OpenAI: автоматично кешує для довгих промптів (без явного API)
MiniMax M3: перевіряй документацію

Економія може бути значною:
- Системний промпт 5000 токенів × 1000 запитів = 5M токенів
- З кешем: тільки перший запит рахується повністю

# Що входить у контекстне вікно?

Не тільки "розмова". У реальних агентах контекст набагато більший:

```
╔══════════════════════════════════════════════════════╗
║  КОНТЕКСТНЕ ВІКНО РЕАЛЬНОГО АГЕНТА                   ║
╠══════════════════════════════════════════════════════╣
║  📋 System Prompt (роль, правила, SOP)   ~1-10K токен║
║  🔧 Tool Definitions (опис інструментів)  ~1-5K токен║
║  📚 RAG-контекст (знайдені документи)    ~2-8K токен ║
║  👤 Профіль користувача                  ~0.5-2K токен
║  📄 Поточна сторінка / файл / дані       ~1-20K токен║
║  💬 Conversation History                 ~2-10K токен║
║  🔨 Tool Results (SQL, API, файли)       ~1-10K токен║
╚══════════════════════════════════════════════════════╝
                         РАЗОМ: 8K - 65K токенів
```

<v-click>

→ При 256K контексті (MiniMax M3) — є запас. При 16K (старі моделі) — критично важливо оптимізувати.

</v-click>

# Проблема: контекст росте → дорожче і гірше

```
Запит  1: [sys 5K] + [tools 2K] + [msg ×1]                = ~8K токенів
Запит 10: [sys 5K] + [tools 2K] + [msg ×9] + [results ×9] = ~30K токенів
Запит 50: [sys 5K] + [tools 2K] + [msg ×49]               = ~90K токенів
                                                ↑ кожен раз платимо за все з початку
```

<v-click>

**Дві реальні проблеми:**

| Проблема | Що відбувається |
|----------|-----------------|
| 💸 **Вартість** | Ціна = вхідні токени + вихідні токени. При довгому чаті кожен запит все дорожчий |
| 🤯 **"Lost in the Middle"** | При великому контексті LLM краще пам'ятає початок і кінець, але губить середину (arxiv:2307.03172) |

</v-click>

<v-click>

**Рішення:** стратегії керування контекстом — **Conversation Memory** (Блок 2)

</v-click>

<!--
НОТАТКИ ДОПОВІДАЧА:
"Lost in the middle" — реальний задокументований феномен.
Дослідження Liu et al. 2023 показали: модель звертає більше уваги

Практично: якщо важливий факт в повідомленні #50 з 100 — LLM може його проігнорувати.
Рішення: важливе — в системний промпт (початок) або в останні повідомлення.

Вартість на прикладі MiniMax M3:

Це реальна вартість яку треба враховувати.

KV Cache (попередній слайд) частково вирішує проблему вартості.
Але не проблему "lost in the middle".
