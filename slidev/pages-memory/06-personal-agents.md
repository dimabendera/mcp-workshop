_Агент що знає вас — не просто інструмент_

<!--

інформацію про конкретного користувача між сесіями.

3. User preferences — переваги і стиль юзера

# Entity Memory: факти про людей і речі

**Термін: Entity** — об'єкт з атрибутами (людина, місце, продукт).

```js
// Структура entity в SQLite / JSON:
const entityMemory = {
  "Іван": {
    type: "user",
    city: "Одеса",
    budget: "8000 грн/міс",
    preferences: ["балкон", "метро"],
    lastSeen: "2024-06-15"
  },
  "ЖК Перлина": {
    type: "property",
    city: "Одеса",
    priceFrom: 7500,
    rooms: 2
  }
};
```

::right::

**При новій сесії** — підвантажуємо в промпт:

```js
const user = entityMemory["Іван"];
const systemPrompt = `
  Юзер: ${user.city}
  Бюджет: ${user.budget}
  Переваги: ${user.preferences.join(", ")}
`;
// Агент одразу знає контекст юзера ✅
```

**LangChain** — автовитяг entities з розмови:

```js
import { EntityMemory } from "langchain/memory";
const memory = new EntityMemory({ llm });
// LLM сам знаходить "Іван", "Одеса"
// і зберігає факти про них автоматично
```

<!--
НОТАТКИ ДОПОВІДАЧА:
Entity extraction — LLM сам визначає ЩО важливо в тексті і зберігає структуровано.

Як це працює під капотом:
При кожному повідомленні → LLM аналізує текст → витягує нові факти про entities
→ оновлює сховище → при наступному запиті підвантажує релевантні entities.

Мінус: LLM може помилитись у тому що "важливо".
Наприклад: "Я їду до брата в Харків" → зберегти "Харків" як місто юзера? Ні!

Для критичних продуктів: явне збереження через UI ("оновити профіль") краще ніж авто.

Зберігання: SQLite або PostgreSQL jsonb колонка.
Для пошуку: якщо entities багато — векторна БД для semantic search.
-->

# Temporal Memory: "коли" щось сталось

**Термін: Temporal Memory** — пам'ять з прив'язкою до часу.

```js
const memoryWithTime = {
  userId: "ivan_123",
  facts: [{
    content: "Шукає квартиру в Одесі",
    createdAt: "2024-01-15",
    lastSeen: "2024-06-10",
    confidence: 0.95
  }]
};
```

Агент знає: факт від 6 місяців тому → може бути неактуальний.

::right::

**Стратегія:** якщо факт старший N місяців → переспитати.

```js
const isStale = (fact) =>
  Date.now() - new Date(fact.lastSeen)
  > 180 * 24 * 60 * 60 * 1000; // 180 днів
```

**Навіщо потрібно:**
- Переваги міняються → старі факти неактуальні
- "Це 6 міс. тому, ви ще шукаєте?"
- Аналіз трендів: "Іван активніший у вихідні"

<!--
НОТАТКИ ДОПОВІДАЧА:
Temporal memory — часто ігнорована але важлива фіча.

Без часових міток → агент може нагадувати юзеру про речі які вже нерелевантні.
"Знайшли квартиру?" → "Так, ще рік тому!" → поганий UX.

Zep (наступний слайд) має вбудовану temporal memory.
Для простої реалізації: просто додай createdAt і updatedAt до кожного факту.
PostgreSQL: jsonb з GIN індексом або окрема таблиця user_memory.

Архівація: факти старші 1 року → переносимо в "архів" і не включаємо в активний контекст.
Але зберігаємо для аналітики.
-->

# Mem0: автоматична екстракція пам'яті

**Mem0** — open source (Apache 2.0) memory layer.
Self-hosted або managed cloud.

```js
import MemoryClient from "mem0ai";
const client = new MemoryClient({ apiKey: process.env.MEM0_API_KEY });

// Mem0 сам витягує факти:
await client.add([
  { role: "user",      content: "Я Іван, шукаю квартиру в Одесі" },
  { role: "assistant", content: "Зрозумів! Є 2-кімнатні варіанти..." }
], { userId: "ivan_123" });

// Semantic пошук:
const r = await client.search("квартира Одеса", { userId: "ivan_123" });
// → [{ memory: "Іван шукає квартиру в Одесі", score: 0.95 }]
```

::right::

# Zep: пам'ять з knowledge graph

**Zep** (Apache 2.0) — спеціалізований memory service.
Entity extraction + Knowledge Graph + temporal.

```js
import { ZepClient } from "@getzep/zep-cloud";
const client = new ZepClient({ apiKey: process.env.ZEP_API_KEY });

await client.memory.add("ivan_session_001", { messages: [
  { role: "user", content: "Привіт, я Іван" },
] });

const r = await client.memory.searchSessions({
  text: "ім'я користувача",
  userId: "ivan_123", limit: 3
});
// Knowledge Graph: Іван ──[шукає]──▶ Одеса
```
