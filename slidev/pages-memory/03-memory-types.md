# Аналогія: пам'ять людини vs пам'ять агента

Когнітивна наука виділяє кілька типів пам'яті у людей. Дослідники AI взяли цю класифікацію.

```
╔══════════════════════════════════════════════════════════════════╗
║  ЛЮДСЬКА ПАМ'ЯТЬ              АНАЛОГ В AI-АГЕНТІ               ║
╠══════════════════════════════════════════════════════════════════╣
║  Сенсорна (секунди)          Поточний вхід + поточний токен    ║
║  • Що ви зараз бачите/чуєте  • Рядок тексту що обробляється    ║
╠══════════════════════════════════════════════════════════════════╣
║  Робоча / Короткострокова    Контекстне вікно (Context Window) ║
║  • "тримати в голові" зараз  • Поточний масив messages         ║
║  • Ліміт: ~7±2 "чанки"       • Ліміт: N токенів               ║
╠══════════════════════════════════════════════════════════════════╣
║  Довгострокова (роки)        Зовнішня база даних               ║
║  • Факти про світ            • Векторна БД, SQL, файли         ║
║  • Спогади про події         • Embedding-пошук, RAG            ║
║  • Як виконувати дії         • Правила, SOP, шаблони           ║
╚══════════════════════════════════════════════════════════════════╝
```

<!--
НОТАТКИ ДОПОВІДАЧА:
Сенсорну пам'ять у LLM можна ігнорувати — це внутрішня механіка трансформера.
Фокус на: Робоча (контекст) + Довгострокова (зовнішнє сховище).

Чому аналогія важлива?
Вона дає інтуїцію: "робоча пам'ять" обмежена (як у людини 7±2 "чанки") →
LLM так само обмежений N токенами контексту.

"Довгострокова" = зовнішня БД → потребує retrieval (пошуку) щоб потрапити в "робочу".
Людина так само: щоб "пригадати" — відбувається retrieval з довгострокової в робочу.
-->

# 4 типи довгострокової пам'яті агента

Довгострокова пам'ять агента ділиться на 4 підтипи:

<div grid="~ cols-2 gap-6">
<div>

**📚 Семантична (Semantic)**
_"Знання про світ"_

```
"SQL JOIN об'єднує таблиці"
"user_id є в таблиці users"
"clickhouse швидший для агрегацій"
```
→ Facts, knowledge, domain info
→ Зберігається: векторна БД, SQL

**📖 Процедурна (Procedural)**
_"Як щось робити"_

```
"При помилці auth → перевір токен"
"SQL запити тільки SELECT"
"Для аналітики → ClickHouse"
```
→ Rules, SOPs, instructions
→ Зберігається: файли, БД

</div>
<div>

**🎬 Епізодична (Episodic)**
_"Що сталось колись"_

```
"Вчора Іван питав про оренду"
"Цей запит давав помилку OOM"
"Минулого разу Join не спрацював"
```
→ Past interactions, examples
→ Зберігається: векторна БД, SQL

**⚡ Кеш (Cache / Working)**
_"Що вже обраховано"_

```
"Цей SQL вже виконувався → результат"
"Embeddings вже порахованo"
"Tool output за годину ще актуальний"
```
→ Computation cache
→ Зберігається: Redis, RAM, pickle

</div>
</div>

НОТАТКИ ДОПОВІДАЧА:
Ця класифікація — теоретична, запозичена з когнітивної психології.
На практиці межі між типами розмиті — і це нормально.

- Семантична: відповідь "що таке X" → факти, знання
- Процедурна: відповідь "як робити X" → правила, SOP
- Епізодична: відповідь "що сталося колись" → historical examples
- Кеш: відповідь "це вже обраховано" → не рахуй знову

Практичний приклад для AI Data-Assist (DIM.RIA):
- Семантична: схема БД, опис колонок, бізнес-логіка
- Процедурна: CLAUDE.md з правилами SQL для ClickHouse
- Епізодична: попередні успішні запити як few-shot приклади
- Кеш: результати популярних агрегацій за останню годину

# Яку пам'ять куди зберігати?

```
╔═══════════════════╦══════════════════╦═══════════════════════════╗
║ Тип пам'яті       ║ Сховище          ║ Як дістати                ║
╠═══════════════════╬══════════════════╬═══════════════════════════╣
║ Семантична        ║ Векторна БД      ║ Similarity search         ║
║ (факти/знання)    ║ (Chroma, Pinecone)║ "Знайди схоже до запиту" ║
╠═══════════════════╬══════════════════╬═══════════════════════════╣
║ Процедурна        ║ Markdown-файли   ║ Keyword або vector пошук  ║
║ (правила/SOP)     ║ або БД           ║ або завжди в промпті      ║
╠═══════════════════╬══════════════════╬═══════════════════════════╣
║ Епізодична        ║ SQL + Векторна   ║ SQL за датою/юзером       ║
║ (минулі події)    ║ (hybrid search)  ║ + similarity для схожих   ║
╠═══════════════════╬══════════════════╬═══════════════════════════╣
║ Кеш               ║ Redis / RAM      ║ Ключ → значення           ║
║ (обраховані дані) ║ / pickle         ║ TTL (час до інвалідації)  ║
╚═══════════════════╩══════════════════╩═══════════════════════════╝
```

<v-click>

**Золоте правило:** не все має потрапляти в промпт одразу.
Агент **шукає** потрібну пам'ять за запитом → вставляє тільки релевантне.

</v-click>

<!--
НОТАТКИ ДОПОВІДАЧА:
Ключова ідея: retrieval-augmented = знаходимо потрібне, а не кладемо все в промпт.
RAG (Retrieval Augmented Generation) — про це в Блоці 4.

Золоте правило (повторити голосно):
"Не кладіть все в системний промпт! Шукайте потрібне і вставляйте тільки це."

Системний промпт = обмежений ресурс.
Якщо туди потрапить 500 сторінок документації → 50K+ токенів → дорого і повільно.
RAG: → знаходимо 5 найбільш релевантних сторінок → 2K токенів → дешево і точно.

# Короткострокова vs Довгострокова

```
╔══════════════════════════╦══════════════════════════════════════╗
║ КОРОТКОСТРОКОВА          ║ ДОВГОСТРОКОВА                        ║
║ (In-context / Working)   ║ (External storage)                   ║
╠══════════════════════════╬══════════════════════════════════════╣
║ Де: контекстне вікно LLM ║ Де: файли, БД, векторна БД          ║
║ Коли: поточна сесія      ║ Коли: між сесіями, постійно          ║
║ Ліміт: N токенів         ║ Ліміт: практично необмежено          ║
║ Швидкість: мгновенна     ║ Швидкість: потрібен retrieval (пошук)║
║ Вартість: токени         ║ Вартість: сховище + retrieval-запит  ║
╠══════════════════════════╬══════════════════════════════════════╣
║ Приклад:                 ║ Приклад:                             ║
║ "У цій розмові юзер      ║ "Цей юзер завжди питає про оренду   ║
║  згадав про квартиру"    ║  в Одесі" (з 500 сесій)             ║
╚══════════════════════════╩══════════════════════════════════════╝
```

<v-click>

**Ключовий патерн:** довгострокова → retrieval → вставляємо в короткострокову → LLM відповідає

```
[Зовнішня БД] --пошук→ [Релевантні фрагменти] --вставка→ [Контекст LLM]
```

</v-click>

<!--
НОТАТКИ ДОПОВІДАЧА:
Цей патерн — основа RAG (Retrieval Augmented Generation).
Блок 4 — детально про RAG: як будувати embeddings, які типи RAG існують, Graph RAG.

Діаграма внизу — ключова для розуміння:
[Зовнішня БД] → пошук за запитом → [3-5 фрагментів] → вставка в контекст → [LLM відповідає]

Це і є "Retrieval Augmented Generation":
- Retrieval = пошук (retrieve relevant docs)
- Augmented = доповнення контексту (augment the prompt)
- Generation = відповідь LLM (generate answer)
-->
